## [Unreleased]

### Removed
- **BREAKING** — `format_deprecated_aliases_block()` is gone, and with it the deprecated `--moss-*` token aliases it emitted into `@layer tokens`. These forwarded the v1.2 names to their v1.3 replacements for one release, as promised when the tokens were renamed; that release has passed. Migrate theme CSS to: `--moss-font-{2xs,xs,sm,lg,xl,2xl,3xl}` → `--moss-size-{…}` (same scale, same values), `--moss-font-weight` → `--moss-font-weight-body`, `--moss-accent-hover` → `--moss-color-accent-hover`, `--moss-text-secondary` → `--moss-color-text-secondary`. A `.moss/theme/style.css` still referencing an old name no longer resolves it: the declaration falls back to the property's inherited or initial value, so the affected text silently renders at the browser default rather than the themed one. `moss describe --json` lists every token that now exists. **Breaking API change: bump the minor (0.2.0 → 0.3.0) and update the workspace `moss-core` version constraint.**
- **BREAKING** — `PageKind::Asset` is gone. It described a synthetic page generated per non-markdown file (the per-image pages an image-only folder produced). That generator was removed in 2026-07 because it could not tell an illustrations folder from a gallery — a folder of 83 article illustrations became 83 standalone pages titled with raw UUIDs. Nothing constructs the variant any more, so the enum no longer advertises a state that cannot occur, and `is_listable_at_depth_all` / `is_listable_at_depth_direct` lose an arm each. `PageKind` is not serialized into any on-disk artifact, so there is no stored data to migrate. **This is a breaking API change: the next release must bump the minor (0.1.x → 0.2.0) and update the workspace `moss-core` version constraint.**

### Added
- Responsive-image ladder vocabulary in `asset_paths`. `ladder_rungs(natural_w, natural_h, is_animated) -> &'static [u32]` is the shared pure oracle that decides which width variants exist for a source; `deployed_width(natural_w, natural_h)` returns the width the base variant actually ships at (the longest edge capped to `DEPLOY_MAX_EDGE`); `to_webp_rung(source, width)` derives a rung URL (`photo.jpg` + 800 → `photo.w800.webp`); `DEPLOY_MAX_EDGE` (2400) and `LADDER` (`[800, 1600]`) are the single source of truth the encode pipeline's `ImageCompressionConfig::default()` also reads. Both `ladder_rungs` and `deployed_width` are **height-aware** on purpose: the encoder caps the longest *edge*, not the width, so a portrait source's deployed base and its rungs are computed from `max(w, h)` — passing width alone silently over-promises rungs for tall images. The function is deterministic in scan-derived inputs only (dimensions + animated flag), never encode/cache state — that constraint is what lets the registration loop, the encode worker, and the synthesizer agree by construction (ADR-013: never emit an unregistered `<source>`). `is_ladder_source_ext(ext)` and `is_webp_source_ext(ext)` are the single extension gates shared across the six pipeline census sites and the synthesizer.
- `contract::sizes`: per-render-context `sizes=` attribute table (`SIZES_FULL_BLEED` = `100vw`, `SIZES_BODY`, `SIZES_CARD`, `SIZES_GALLERY`), encoding the default theme's layout widths off `site.css` tokens. Part of the theme-author contract surface; no override machinery yet (a theme that changes layout gets slightly-suboptimal, never-broken fetches). Lives in a new consts-only module beside `tokens`/`components` under `contract/` (not inside `components.rs`).
- `AssetSnapshot::animated` (a `HashMap<PathBuf, bool>`) and its `is_animated(&PathBuf) -> bool` accessor (missing key reads `false`). The synthesizer consults it to suppress the responsive ladder for animated sources (multi-frame GIF / animated WebP), which must never be re-encoded into still rungs. Populated from scan metadata's new `is_animated` flag.
- LaTeX math parsing, off by default. `ParseConfig::math` gates pulldown-cmark's `ENABLE_MATH`, so `$…$` and `$$…$$` become `Inline::Other` nodes rendering the equation's own source — **`$` delimiters included** — HTML-escaped, in `<code class="moss-math" data-moss-math="inline|display">`. The delimiters are part of the payload because P1 has no typesetting engine: the span is what the reader sees, so emitting only the inner TeX would silently delete two characters of the author's prose. That is not hypothetical — `一个$5，两个$10` parses as math (a full-width comma is non-whitespace, so it closes the span) and would have reached readers as `一个5，两个10`. A genuine equation therefore shows a visible `$…$` until a typesetting phase replaces the span. There is no typesetting engine in moss-core (ADR-030 keeps it app-side); this is the honest-fallback layer a renderer later upgrades to SVG. Existing callers are unaffected: `ParseConfig::default()` leaves math off, `$` stays an ordinary prose character, and no committed fixture changes.
  - Enabling math **without** handling both events silently deletes every equation — `Energy $E = mc^2$.` parsed to `<p>Energy .</p>`, because the AST walker's catch-all arms drop unrecognized inline events. The handler arms and the `parse_inline_event` whitelist entry that prevent this are covered by `tests/math_parsing.rs`, including math nested in list items — the wiring path a paragraph-only test cannot see, since `collect_item_blocks` is that whitelist's only caller. Math in table cells, blockquotes, links, callout bodies and footnotes reaches the parser by other routes and is covered separately.
  - `tests/fixtures/math-delimiters.vectors.json` records pulldown's `$` open/close rules as measured golden vectors — the cross-language contract a future editor grammar must agree with. The rule that decides every false positive: a closing `$` must be immediately preceded by a non-whitespace byte. `$5 and $10` survives only because the second `$` follows a *space* — that is a habit of English typography, not a property of ASCII or of currency. Put any punctuation between two prices and the span closes: `$5-$10` (price range), `$5/$10` (tiers) and `$9.99 ($8.99 for members)` (parenthetical) all parse as math, in English prose, on a default site. Unspaced CJK (`一个$5，两个$10`) parses for the same reason — a full-width comma is non-whitespace. In every case the text stays byte-exact, because the fallback keeps the delimiters; what changes is *presentation*, since the run renders in a monospace `<code>` span on published pages and in RSS item content. Any site whose prose puts two `$` in one paragraph without a space before the second should set `math = false` or escape as `\$`.
  - **Upgrade note — headings whose math contains `*`.** `[site].math` defaults to on, so upgrading enables math for sites that never set the key. Heading anchors normally survive that flip untouched, because the slug is computed from the equation's restored source. They do **not** survive when the TeX contains markdown-active characters: with math off, intraword `*` is eaten as emphasis first, so `# Convolution $f*g$ and $h*k$ end` was published as `convolution-$fg$-and-$hk$-end` and becomes `convolution-$f*g$-and-$h*k$-end`. `# Dual $V^*$ and $W^*$ end` moves the same way, and dual-space notation is common. Where this fires, external deep links, in-page TOC entries and `[[Page#Heading]]` wikilinks pointing at the old anchor break. Set `math = false` to keep the published anchors. See ADR-030 §"Upgrade-time anchor movement".
- `RenderHooks::render_math(out, tex, display, fallback_html)` takes a `fallback_html` argument (P2, ADR-030). moss-core stays engine-free — its default impl emits the escaped-source `<code class="moss-math">` chip — but the app override typesets `tex` to an inline `<svg>` via RaTeX and, when its crash-safety envelope refuses (oversized / deeply-nested / CJK / malformed input), emits `fallback_html` instead. Passing the pre-rendered fallback keeps moss-core from having to know either the engine or the fallback markup. Same-cycle refinement of the unreleased `render_math`; no released signature changed.
- `ast::parser_options(math: bool)`: the single constructor for moss's pulldown-cmark option set (strikethrough, tables, footnotes, wikilinks, plus math on request). Parser construction had drifted across five independent sites; call this instead of assembling `Options` by hand, and where a site genuinely needs a different set, call it and remove the option explicitly so the divergence stays visible.
- `link_completions::rank_completions`: deterministic ranking for wikilink/embed completion candidates. Sort keys: trigger-kind fit, prefix-at-start match quality, same-language-tree as `from_file`, directory-tree proximity, insert-value length, then lexicographic tiebreaks — mirroring `content_graph`'s resolver tiebreak chain so the dropdown surfaces the candidate a `[[link]]` would actually resolve to.
- `resolve::md_extract`: pure markdown reference extractor (`extract_md_references`) returning every wikilink / embed / markdown-link / markdown-image token with byte offsets. Zero I/O; backs rename-with-references and delete-with-references in the editor.
- `frontmatter::strip_control_chars_str`: strips stray C0/C1 control characters (excluding TAB/LF/CR) from a string. Applied at the frontmatter write boundary as defense-in-depth against a macOS Tauri multiwebview bug that can type raw arrow-key control codes into text inputs (`tauri-apps/tauri#10194`); exposed `pub` so other crates' write paths can apply the same strip.

### Fixed
- The implicit-figure `<figcaption>` now renders its alt content as inline markdown (Pandoc's implicit-figure model): `*em*`, `[links](…)`, `` `code` `` and math survive as typed AST nodes instead of being flattened to plain text — so `![before *em* and $x^2$ after](p.png)` captions with `<em>em</em>` and a typeset equation (via `RenderHooks::render_math`) rather than stripped markers and raw `$x^2$` source. The `alt=` attribute is unchanged: it stays the flat plain-text source (math delimiters verbatim) for assistive tech and blocked-image fallback. Wikilink-embed captions (literal pothole alias, not markdown), width-carrying alts (`![cap|50%](p)`), and plain-text alts keep their previous byte-identical output.
- Section-scoped embeds with a human-readable anchor (e.g. `![[file#Heading With Spaces]]`) now slugify the target before matching, so they inline just the referenced section. Previously the raw anchor was compared against slugified headings, never matched, emitted a spurious `Heading '#...' not found` diagnostic, and fell back to inlining the entire file. Anchors already in slug form are unaffected.

### Changed
- Doc-only: the `asset_paths::ladder_rungs` deterministic-agreement contract now records that the **EXIF-orientation agreement holds unconditionally**. The app-side scan-swap fix (responsive-image-variants design follow-up #1) makes scan store the oriented display dims for EXIF-rotated png/webp/jpeg, so the census sites and the encode side derive the identical ladder for those too — the prior "holds except for EXIF-oriented png/webp" exception (and its `render/image.rs` back-references) is removed. No code or behavior change in this crate.
- Raster-original rendering (`render/image.rs`) now emits a **responsive multi-width `srcset`** instead of the single-URL WebP `<source>`. For png/jpg/jpeg sources the `<picture><source>` gains a ladder — `srcset="photo.w800.webp 800w, photo.w1600.webp 1600w, photo.webp 2000w"` with a context-appropriate `sizes=` — the top descriptor being the base WebP at its deployed (edge-capped) width. For **webp** sources (previously a bare `<img>`) the ladder attaches **directly to the `<img srcset>`** with no `<picture>` wrapper, since the base `src` is already WebP. Sources at or below the first rung (≤800px), animated sources, and unknown-dimension sources fall back to the pre-feature single-URL shape byte-for-byte; `EmailBody`, og:image, logo, favicon and bare-fragment contexts never carry `srcset` and are unchanged. Rung existence is decided by `asset_paths::ladder_rungs` from the same scan-derived dimensions the build pipeline registers and encodes, so every emitted rung URL is a promised (`set_pending`) variant — ADR-013 holds by construction.
- `render/image.rs` ladder `srcset` URLs now percent-encode literal commas (`,` → `%2C`) in each candidate URL (`encode_srcset_url`), so a comma-named source (`a,b.jpg` → `a%2Cb.w800.webp 800w, …`) no longer mis-splits the comma-delimited candidate list in browsers or the html_post rewriter (responsive-image-variants follow-up #4). Only the ladder `srcset` candidates are encoded — the base `<img src>`/legacy single-URL `<source>` keep their literal comma (a single unambiguous URL). A static file server decodes `%2C` → `,` to the on-disk `a,b.w800.webp`. Comma-free filenames are byte-for-byte unchanged.
- **BREAKING** — the heading modules are consolidated under one parent. `moss_core::extract_headings::` is now `moss_core::heading::extract::`, and `moss_core::heading_anchor::` is now `moss_core::heading::anchor::`; both old module paths are gone. The commonly-used items keep working through re-exports — `moss_core::extract_headings`, `moss_core::HeadingInfo` and `obsidian_heading_anchor` are all still reachable via `heading` — so only code that named the old *module* paths must change. Heading logic previously lived in four homes with no single owner of "what is a heading's text?"; P1's math work had to keep the math-on slug, the rendered `<hN id>` and the wikilink graph's raw-line slug in agreement across all of them, and a false anchor invariant survived review partly because no one module owned the answer. Folds into the same minor bump the `PageKind::Asset` removal already requires (0.1.x → 0.2.0).
- `ComponentEntry::is_public()` now also returns `false` for internal implementation classes (`moss-apply*`), hiding them from `moss describe` and `docs/contract/reference.md`.
- Subscribe-form contract: `moss-subscribe-form` now always uses `data-position="inline"` (the auto-injected email footer and the `:::subscribe` shortcode emit identical HTML; footer vs in-page styling keys on the `footer` ancestor). Adds a `data-button-override` attribute, emitted when an author overrides the button label (`:::subscribe{button="…"}`); on such forms `subscribe.ts` leaves both the button label and the placeholder as authored instead of overwriting them with the language default. The former `data-position="footer"` value and the `footer-shape` slot / `data-moss-shape` footer attribute are removed.
